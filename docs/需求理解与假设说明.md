# 需求理解与假设说明

## 关键假设

1. 分级审批金额阈值：`0 <= amount < 1000` 进入一级审批，`amount >= 1000` 进入二级审批。理由是小额异常需要快速处理，高额异常需要更强授权；阈值落在 `approval_rules` 中可配置。
2. 审批超时时长：一级审批 24 小时，二级审批 12 小时。二级审批更高风险但通常人更少，因此设置更短提醒和自动兜底。
3. 重提次数上限：拒绝后最多重提 2 次，超过后自动关闭并要求重新发起工单，避免无效循环。
4. 物流异常类型映射：丢件和破损生成客户理赔与库存回滚；客户拒收触发退货入库；超时未签收触发人工跟进；地址错误触发重新发货，一般不生成赔付。
5. 角色权限划分：操作员可上报和扫描；一级审批人处理小额物流工单；二级审批人处理高额和升级工单；品控主管可做误判快速放行；管理员维护规则。
6. V2 数据同步频率与一致性策略：发起上报和扫描时实时调用 V2；列表和详情展示可使用 10 分钟一次的本地快照。若 V2 与快照不一致，关键动作以 V2 实时结果为准，并记录差异日志。
7. 品控暂扣超时时长：2 小时，独立于审批超时。品控暂扣会造成货物压仓和出库阻塞，因此远短于审批超时；超时后强制进入二级审批。
8. 品控规则触发阈值：数量差异率大于等于 2%、破损等级大于等于 3、规格偏差大于等于 1 个规格档、标签错误任意命中、批次异常任意命中。所有阈值存入 `quality_rules.condition`，不硬编码。
9. 品控主管权限边界：品控主管只能执行误判快速放行和填写复核原因，不能审批自己创建的工单；可与一级审批人兼任，但同一工单仍受“上报人不能自批”约束。

## 产品澄清问题

- 多仓库/多商户是否是当前必须支持的真实场景，还是可以先按单租户实现？
- V2 是否允许新增 `updated_at` 和异常标记字段，还是只能新增只读接口？
- 品控快速放行后是否需要通知 WMS 或 V2，还是 V3 内部解锁即可？
- 赔付金额是否由审批人填写，还是按运单金额、商品金额或差价规则自动计算？
- 二级审批超时后是自动驳回还是转交管理员，本方案先选择自动驳回并留痕。

## 一致性方案

审批通过后的状态变更、库存流水、赔付记录、批次解锁必须在一个数据库事务内完成。每条库存流水和赔付记录都保存 `approval_record_id`，保证可追溯。对于需要调用外部系统的动作，先在 V3 内记录 `pending_external_sync`，由幂等后台任务补偿重试。
