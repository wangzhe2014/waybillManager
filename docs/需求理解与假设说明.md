# 需求理解与假设说明

文档版本：V3 当前实现口径  
更新时间：2026-07-06  
适用项目：运单全流程管理系统 V3

## 1. 文档目的

本文用于说明对《考试要求-文件版本V3》的产品理解、业务边界、状态机设计、关键规则假设和当前实现取舍。

V3 的核心目标不是在 V2 上追加页面，而是建设一个独立的运单全生命周期管理系统。V3 负责扫描品控、物流异常上报、分级审批、执行联动、赔付记录、库存流水、接口监控和定时任务；V2 继续负责 AI 录单解析和运单主数据来源。

本文中的“明确需求”表示题目已有直接要求；“项目假设”表示题目留白但为了完成闭环必须补齐的业务判断；“待确认项”表示上线或进一步产品化前仍需要业务方确认。

## 2. 总体需求理解

### 2.1 系统定位

V3 是独立于 V2 的运单全流程管理系统，覆盖以下链路：

```text
V2 录单解析 -> V3 运单快照 -> 扫描品控 -> 异常工单 -> 分级审批 -> 执行联动 -> 赔付/库存/关闭
```

V3 与 V2 必须独立部署、独立数据库，不允许 V3 直接连接 V2 数据库。所有运单校验和快照补全都通过 V2 HTTP API 完成。

### 2.2 V2 与 V3 的职责边界

| 系统 | 职责 | 数据边界 |
| --- | --- | --- |
| V2 | AI 录单解析、结构化运单主数据、运单/SKU 查询接口 | 拥有运单主数据 |
| V3 | 扫描品控、异常上报、审批、执行联动、赔付、库存流水、监控 | 拥有工单和流程数据 |

项目假设：V3 保存的 `waybill_snapshots` 是只读快照/缓存，不是 V2 运单主表。关键动作前优先实时调用 V2 校验，避免对虚构运单或错误 SKU 创建工单。

### 2.3 两类异常入口

| 异常类型 | 触发方式 | 入口 | 是否自动建工单 | 是否锁批次 |
| --- | --- | --- | --- | --- |
| 物流异常 | 人工上报 | 工单追踪/上报异常 | 是 | 否 |
| 品控异常 | 扫描触发 | 扫描品控 | 命中规则时自动创建或复用 | 是 |

项目假设：物流异常发生在发货后或履约过程中，核心是客户侧履约补偿；品控异常发生在发货前，核心是阻止问题批次继续出库，并对供应商追偿或进行库存处理。

## 3. 业务流程理解

### 3.1 正常扫描流程

1. 仓库人员输入运单号、SKU 编码、批次号。
2. V3 调用 V2 SKU 归属校验接口。
3. 校验通过后，V3 补写或更新运单快照。
4. V3 执行品控规则匹配。
5. 未命中规则时，写入扫描记录，批次状态为“可用/可出库”，不创建工单。

项目假设：正常扫描也必须写入快照，因为 `scan_records.waybill_no` 需要关联真实运单快照；否则 V3 本地没有快照时会出现外键或审计链路缺失。

### 3.2 异常扫描流程

1. V3 调用 V2 校验运单与 SKU 关系。
2. 校验通过后执行品控规则。
3. 命中规则时，批次进入品控暂扣。
4. 系统创建品控异常工单，或复用同一 SKU + 批次下仍未关闭的品控工单。
5. 写入扫描记录，关联命中的规则和工单。
6. 品控工单默认进入规则指定的审批层级，常见高风险规则直接进入二级审批。

项目假设：同一 SKU + 批次存在未关闭品控工单时，重复扫描只追加扫描记录，不重复创建工单，避免同一实物批次被多个工单重复锁定。

### 3.3 物流异常上报流程

1. 操作员选择异常类型并填写异常信息。
2. V3 调用 V2 运单详情接口验证运单真实存在。
3. 校验成功后写入或更新运单快照。
4. 创建物流异常工单。
5. 根据金额审批规则进入一级或二级审批。

项目假设：物流异常必须先通过 V2 实时校验，不允许为了演示创建不关联真实运单的工单。

### 3.4 审批与执行联动流程

1. 审批人只能处理自己权限范围内的工单。
2. 上报人不能审批自己提交的工单。
3. 审批通过后，系统自动执行联动动作，而不是要求用户手动点“执行联动”。
4. 联动完成后工单进入已完成。
5. 审批拒绝后，工单进入驳回状态，可由上报人重新提交，受次数上限约束。

项目假设：`/api/tickets/:ticketId/execute` 保留为补偿/重试入口，主流程不依赖人工手动执行。审批通过后的赔付、库存、批次状态变化应尽量在同一事务内完成。

## 4. 状态机理解

### 4.1 工单状态

当前的 V3 工单状态采用以下枚举：

| 状态 | 中文含义 | 说明 |
| --- | --- | --- |
| `pending_review` | 待审批 | 新建后等待进入审批或被规则分派 |
| `level1_reviewing` | 一级审批中 | 小额或一般物流异常审批 |
| `level2_reviewing` | 二级审批中 | 高额、升级或高风险品控异常审批 |
| `rejected` | 已驳回 | 可按规则重新提交 |
| `executing` | 执行中 | 审批通过后执行联动的中间态 |
| `completed` | 已完成 | 联动完成或快速放行完成 |
| `closed` | 已关闭 | 超时兜底或重提次数耗尽后关闭 |

明确需求：工单状态机必须覆盖分级审批、拒绝重提、超时自动流转、执行联动和完成关闭。

项目假设：`executing` 是内部联动中间态。当前系统审批通过后会自动执行联动并尽快落到 `completed`，页面侧通常不会长期停留在执行中。

### 4.2 批次状态

当前 V3 批次/扫描状态采用以下枚举：

| 状态 | 中文含义 | 说明 |
| --- | --- | --- |
| `available` | 可用 | 扫描通过，可继续出库 |
| `qc_hold` | 品控暂扣 | 命中异常规则，禁止出库 |
| `qc_released` | 已放行 | 快速放行或审批后放行 |
| `returned_supplier` | 已退供 | 退回供应商 |
| `repurchasing` | 重采中 | 原批次作废或需补采 |
| `downgraded` | 已降级 | 降级处理后出库或处置 |

明确需求：品控批次状态机必须与工单状态机分离，二者通过扫描记录和工单关联。

项目假设：批次状态变化只由品控扫描、品控主管快速放行或审批通过后的质量处理动作触发，不能随着工单状态普通变化自动解锁。

## 5. 规则与阈值假设

### 5.1 审批金额规则

当前默认规则：

| 规则 | 金额范围 | 目标审批层级 |
| --- | --- | --- |
| 小额一级审批 | `0 <= amount <= 999.99` | 一级审批 |
| 大额二级审批 | `amount >= 1000` | 二级审批 |

项目假设：1000 元作为默认阈值，兼顾处理效率和风险控制。实际业务可在规则配置页面维护，不应写死在业务代码中。

### 5.2 品控规则

当前默认规则覆盖：

| 规则编码 | 规则名称 | 异常子类型 | 默认审批层级 |
| --- | --- | --- | --- |
| `QR-DAMAGE-03` | 外观破损暂扣 | 外观破损 | 二级审批 |
| `QR-QTY-DIFF-02` | 数量不符暂扣 | 数量不符 | 一级审批 |
| `QR-LABEL-01` | 标签错误暂扣 | 标签错误 | 一级审批 |
| `QR-BATCH-01` | 批次异常暂扣 | 批次异常 | 二级审批 |
| `QR-SPEC-01` | 规格不符暂扣 | 规格不符 | 一级审批 |

项目假设：品控规则应可配置，包括规则名称、规则编码、异常类型、严重程度、条件字段、操作符、阈值、启用状态和进入审批层级。

### 5.3 重新提交次数

项目假设：驳回后最多允许重提 2 次，超过后关闭或要求重新发起工单。理由是异常单据需要允许修正材料，但不能无限循环占用审批资源。

### 5.4 超时规则

当前定时任务配置：

| 任务 | 路径 | 频率 | 行为 |
| --- | --- | --- | --- |
| 超时自动流转 | `/api/timeouts/process` | 每 5 分钟 | 待审批/一级超时升级二级，二级超时自动驳回并关闭 |

项目假设：当前部署环境不使用 Vercel Cron，改由 GitHub Actions Schedule 作为外部调度器，每 5 分钟调用 V3 的 `/api/timeouts/process`。V3 服务端使用 `CRON_SECRET` 或 `TIMEOUT_CRON_SECRET` 做接口鉴权，GitHub Actions 侧通过仓库 Secrets 配置 `V3_TIMEOUT_URL` 和 `CRON_SECRET`，不在代码中保存真实密钥。

## 6. 执行联动假设

### 6.1 物流异常动作映射

| 异常类型 | 默认动作 | 赔付方向 | 库存影响 |
| --- | --- | --- | --- |
| 丢件 | 客户赔付 | 赔付客户 | 无直接库存流水 |
| 破损 | 客户赔付 | 赔付客户 | 无直接库存流水 |
| 客户拒收 | 退货入库 | 无默认赔付 | 入库 |
| 超时未签收 | 重新发货 | 无默认赔付 | 出库 |
| 收货地址错误/地址错误 | 重新发货 | 无默认赔付 | 出库 |

业务理由：丢件、破损通常以客户体验和货损理赔为主；客户拒收优先回收入库；地址错误和超时未签收更常见的履约动作是重新发货。

### 6.2 品控异常动作映射

| 异常类型 | 默认动作 | 赔付方向 | 批次/库存影响 |
| --- | --- | --- | --- |
| 数量不符 | 重新采购 | 向供应商追偿 | 重采中/状态变更 |
| 外观破损 | 退回供应商 | 向供应商追偿 | 退供/出库 |
| 规格不符 | 重新采购 | 向供应商追偿 | 重采中/状态变更 |
| 标签错误 | 降级处理 | 向供应商追偿 | 降级 |
| 批次异常 | 退回供应商 | 向供应商追偿 | 退供/出库 |

业务理由：品控问题发生在出库前，责任通常在供应链或供应商侧，赔付方向应与物流异常区分，使用“向供应商追偿”。

### 6.3 手动执行入口

项目假设：审批通过后自动执行联动是主流程；手动执行接口仅用于补偿、重试和排障，不作为普通业务按钮暴露给操作人员。

## 7. 数据模型理解

V3 独立数据库至少包含以下核心表：

| 表 | 作用 |
| --- | --- |
| `waybill_snapshots` | V2 运单只读快照 |
| `integration_logs` | V2 接口调用和定时任务日志 |
| `exception_tickets` | 异常工单 |
| `approval_records` | 审批记录 |
| `compensation_records` | 赔付记录 |
| `inventory_batches` | V3 批次库存 |
| `inventory_movements` | 库存流水 |
| `scan_records` | 扫描记录 |
| `quality_rules` | 品控规则 |
| `approval_rules` | 审批规则 |
| `ticket_events` | 工单事件轨迹 |

项目假设：V3 的库存表是流程侧库存/批次状态账，不替代真实 WMS。若未来接入 WMS，应通过接口同步库存动作，并保留 V3 本地流水作为审计账。

## 8. 角色与权限假设

当前 V3 采用页面身份切换模拟角色：

| 角色 | 权限边界 |
| --- | --- |
| 操作员 | 扫描品控、上报物流异常 |
| 一级审批人 | 审批一级工单 |
| 二级审批人 | 审批二级工单 |
| 品控主管 | 误判快速放行、处理品控主管专属动作 |
| 执行专员 | 查看执行结果、作为后续执行岗位预留 |

项目假设：考试/演示阶段用身份切换模拟登录态；生产环境应接入真实登录、用户、角色和组织权限，不应依赖前端选择器作为安全边界。

## 9. 接口与监控假设

### 9.1 V2 接口依赖

V3 依赖 V2 提供：

| 接口能力 | 用途 |
| --- | --- |
| 运单详情查询 | 物流异常上报前校验运单并补快照 |
| SKU 归属校验 | 扫描品控前校验运单与 SKU 关系 |
| 增量运单同步 | 可选，用于后续快照刷新 |

项目假设：V2 API 使用服务间密钥鉴权；V3 记录每次调用的 requestId、接口、状态、耗时和错误，方便排查跨系统问题。

### 9.2 生产与本地网络

项目假设：生产部署环境应能直接访问 V2 生产域名。本地开发如果因为网络或 DNS 无法直连 Vercel 域名，可以通过本机代理启动 V3，但这只属于本地联调手段，不应写入生产配置。

### 9.3 接口监控

接口监控页面用于查看 V2 调用和后台任务日志，支持按 requestId、接口路径搜索和分页。状态需以中文展示，避免业务用户看到纯英文状态而无法判断含义。

## 10. 页面功能理解

| 页面 | 需求理解 |
| --- | --- |
| 总览 | 展示核心指标、关键工单、接口成功率，避免假数据和刷新追加错觉 |
| 扫描品控 | 执行单次扫描，突出必填项和错误提示，只展示最近结果 |
| 扫描记录 | 查询历史扫描记录，支持分页和条件查询 |
| 工单追踪 | 查询、上报、查看详情，异常类型使用枚举 |
| 分级审批 | 按当前身份展示可审批工单，支持通过、拒绝、快速放行 |
| 赔付记录 | 查看审批联动生成的客户赔付和供应商追偿 |
| 库存流水 | 查看重新发货、退货入库、品控处理产生的库存流水 |
| 规则配置 | 维护审批规则和品控规则，支持搜索、分页、新建、编辑、启停 |
| 定时任务 | 独立展示后台任务配置、手动执行、跳转日志 |
| 接口监控 | 查询 V2 接口和定时任务调用日志 |

## 11. 当前实现取舍与限制

1. 扫描记录查询当前支持筛选与分页，但 Supabase 场景下部分筛选采用“取最近 500 条后内存过滤”的实现。小规模验收可用，生产应改为数据库侧条件查询并补充索引。
2. 页面身份切换是演示级角色模拟，不是生产级认证授权。
3. V3 库存是流程审计账，不是完整 WMS 库存系统。
4. 生产定时任务依赖外部调度器正确配置目标地址和密钥；当前推荐 GitHub Actions Secrets：`V3_TIMEOUT_URL`、`CRON_SECRET`。
5. AI 建议审批属于题目可选加分项，当前核心闭环不依赖 AI，避免 AI 不稳定阻塞主流程。
6. 本地使用生产 V2 域名时可能依赖代理；线上部署应以服务器到服务器直连为准。

## 12. 待业务确认项

1. 审批金额阈值 1000 元是否符合真实业务授权标准。
2. 驳回后最多重提 2 次是否合理。
3. 一级、二级审批超时时长是否需要区分工作日、节假日和班次。
4. 二级审批超时后自动驳回关闭是否符合业务风险偏好，还是应转交管理员。
5. 各物流异常到执行动作的默认映射是否符合实际 SOP。
6. 各品控异常到退供、重采、降级、放行的默认映射是否符合仓库和采购流程。
7. 赔付金额是否完全按运单金额、SKU 金额、人工填写金额，还是按异常类型自动计算。
8. V3 库存流水是否需要同步真实 WMS，或仅作为流程留痕。
9. 品控主管快速放行是否需要二次确认或照片/附件证据。
10. 生产环境是否需要多仓、多店铺、多租户权限隔离。

## 13. 验收口径

项目达到“需求符合”的最低标准应包括：

1. V3 与 V2 独立部署、独立数据库，V3 不直连 V2 数据库。
2. 物流异常上报必须校验真实 V2 运单。
3. 扫描品控必须校验 V2 运单与 SKU 关系。
4. 正常扫描和异常扫描都能写入扫描记录。
5. 品控异常能创建或复用工单，并锁定批次。
6. 分级审批、拒绝重提、快速放行、超时流转受状态机约束。
7. 审批通过后自动执行赔付、库存或批次联动。
8. 赔付记录区分“赔付客户”和“向供应商追偿”。
9. 库存流水可追溯到工单和审批记录。
10. 接口监控能追踪 V2 调用和定时任务执行。
11. 规则配置可维护审批规则和品控规则，不依赖硬编码。
12. 页面查询、分页、错误提示和状态中文展示满足业务用户操作。

## 14. 总结

本项目的核心理解是：V3 是一个围绕异常处理和履约风控建设的独立流程系统。它不替代 V2 的录单能力，也不直接篡改 V2 主数据，而是通过接口校验和快照机制建立自己的流程账。

业务设计上，物流异常和品控异常必须区分责任方向、触发方式和库存影响；技术设计上，工单状态机和批次状态机必须分离，并在审批通过后的执行联动中保持一致性。题目中未给出的金额阈值、超时时长、重提次数和动作映射，本文已给出当前项目假设，后续可根据真实业务 SOP 调整为后台配置。
