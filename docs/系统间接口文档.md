# V3 调用 V2 系统间接口文档

## 原则

V3 与 V2 独立部署、独立数据库。V3 不直连 V2 Supabase，只通过 HTTP API 获取运单和 SKU 数据。每次调用都携带 `Authorization` 与 `X-Request-ID`，V3 将调用结果写入 `integration_logs`。

## 鉴权

```http
Authorization: Bearer <V2_API_KEY>
X-Request-ID: req_20260703_094210_7b1c
```

V2 侧新增接口时只校验服务间 API Key，不引入复杂 OAuth。后续如多租户扩展，可在 Token 中加入 `tenant_id` 或使用额外的 `X-Tenant-ID`。

## 接口列表

### 1. 获取运单详情

`GET /api/v3/shipments/:externalCode`

用于 V3 发起异常上报前实时校验运单存在，并获取 SKU 明细生成本地快照。

响应示例：

```json
{
  "waybillNo": "PS2512220005001",
  "storeName": "海口龙湖天街店",
  "receiverName": "林晓",
  "receiverPhone": "13800002190",
  "receiverAddress": "海南省海口市龙华区龙湖天街",
  "amount": 2680,
  "skus": [
    {
      "skuCode": "ZBWP10086",
      "skuName": "冷链牛肉卷",
      "skuQuantity": 4,
      "skuSpec": "500g"
    }
  ]
}
```

### 2. 校验 SKU 归属

`GET /api/v3/shipments/:externalCode/skus/:skuCode/validate`

用于扫描录入时确认 SKU 属于指定运单，避免扫描到无关货物。

响应示例：

```json
{
  "valid": true,
  "waybillNo": "PS2512220005001",
  "skuCode": "ZBWP10086",
  "skuName": "冷链牛肉卷"
}
```

### 3. 增量同步运单列表

`GET /api/v3/shipments/sync?updatedAfter=2026-07-03T00:00:00.000Z&limit=200`

用于 V3 初始化或定时刷新本地快照。V2 当前 `shipments` 表没有 `updated_at`，短期可按 `created_at` 增量，正式二开时建议补 `updated_at` 字段。

### 4. 异常标记回写，选做

`POST /api/v3/shipments/:externalCode/exception-marker`

用于 V3 将“存在未关闭异常”回写给 V2 展示。此接口为加分项，不影响 V3 主流程。

## 超时与重试

- 单次超时：3000ms。
- 重试：最多 2 次，只重试 GET 或带幂等键的 POST。
- 降级：V2 不可用时，工单详情可展示本地快照，并明确标注“使用本地缓存，同步于 XX 时间”。发起新异常和扫描归属校验属于关键真实性动作，V2 不可用时不允许创建新工单。

## 老系统二开策略

如果 V2 没有现成接口，新增 `/api/v3/*` 路由，不改动原 `/api/shipments` 行为，避免破坏已有导入系统。字段采用向后兼容策略：新增字段只增不删，金额统一以 decimal 字符串或 JSON number 返回，V3 侧用 decimal/NUMERIC 存储。
